## 6b. H8 (double execution / question-answer desync) — CONFIRMED reproducible, root cause understood, FIX DEFERRED (needs care)

### Iteration 3 update (continuation after the H3 fix + concurrent-workspace incident described in campaign-log.md ITERATION 2)

Live trace `traces/golden-trace-rapidfire-desync.mjs` fired 3 DISTINCT questions (each about a uniquely-identifiable fact in the same reference doc) through the real `__e2e__:ask`/WTA path in overlapping flight (no await between sends). **Reproduced exactly the founder's "answers a completely different question" symptom**: question 1 ("database migration deadline") received the answer to question 3 ("wifi password"); questions 2 and 3 received nothing (`noDecision: true`).

Root-caused via console capture (`traces/golden-trace-rapidfire-console.mjs`):
1. `[IntelligenceEngine] Planner stayed silent { reason: 'cooldown' }` fired for 2 of the 3 overlapping triggers — the live auto-trigger's cooldown gate rejects rapid-fire triggers by design (this itself is correct, expected behavior for the AUTO-TRIGGER path).
2. Only ONE trigger actually produced an answer. But `IntelligenceEngine.ts`'s `emit('suggested_answer', finalWtaAnswer, question, confidence)` call (~line 2439) is explicitly commented **"The suggested_answer emit below is UNGATED"** — i.e. deliberately NOT checked against `this.currentGenerationId` before firing, unlike nearly every other checkpoint in the same method (`generationId = ++this.currentGenerationId` at line 1731, checked at lines 1850/1874/1884/1949/2141 etc.). The comment explains why: "that is the live UI delivery the user still needs to see; only storage is gated" — i.e. this was a DELIBERATE prior fix to prevent the app going silent when a generation gets superseded (this matches a past fix pattern: mirrors "What to answer stops responding after a few messages" P0, referenced in `ipcHandlers.ts:7754-7757` for the sibling cooldown-skip fix on manual button presses).
3. Confirmed this is REACHABLE in real production, not just the E2E harness: `ipcHandlers.ts`'s `generate-what-to-say` handler (manual button/hotkey press, ~line 7747) explicitly sets `skipCooldown: true` specifically so a manual press is never throttled — meaning a user's manual press CAN genuinely overlap with an in-flight auto-trigger-initiated generation for a DIFFERENT question. The production listener in `main.ts:5629-5634` forwards every `suggested_answer` emit verbatim to the renderer with `{answer, question, confidence}` but no generation id.
4. The renderer's `finalizeStreamingByIntentMessages` (`src/lib/overlayMessagePersistence.mjs:27`) guards against cross-INTENT clobbering (e.g. a stray WTA answer landing on an open manual-chat placeholder) via `streamingMsgId`/intent matching, but has NO mechanism to reject a `what_to_answer` answer that belongs to an OLDER, superseded question when a NEWER `what_to_answer` generation is also in flight — it just finalizes whatever answer arrives into the currently-open same-intent streaming row, without checking whether that answer's `question` field matches the row's actual question.

### Why this fix is DEFERRED, not landed this iteration

The "ungated" comment is not an oversight — it is a documented prior fix for a real regression (silent app / no answer shown). A naive fix (gate the emit on `currentGenerationId === generationId`, dropping stale-generation answers) would very likely REINTRODUCE that already-fixed bug: a superseded generation would go silent again exactly when the cooldown/speculative-reuse logic decides NOT to restart cleanly. Per R2 (anti-thrash), this needs a design that satisfies BOTH constraints simultaneously — never silently drop an answer AND never attach the wrong answer to the wrong question — which most likely means: keep the emit ungated (always deliver SOMETHING), but attach the actual `generationId`/originating-question identity to the emitted event so the RENDERER can make an informed choice (e.g. render a stale-but-superseded answer as its own new turn tagged to its real question, rather than overwriting the currently-open row for a different, newer question). This is a two-file coordinated change (emit signature in `IntelligenceEngine.ts` + preload/IPC channel + renderer's `finalizeStreamingByIntent`) and deserves its own dedicated fix-and-verify iteration rather than a rushed patch. Pinned as the top Phase 1 priority for the NEXT iteration.

### Verdict: **H8 CONFIRMED as a code-level defect, but REVISED to REFUTED as a real-production defect** (iteration 4 correction — see below)

**CORRECTION (iteration 4):** iteration 3's claim that this is "reachable in real production via a manual press racing an auto-trigger" was WRONG — an overclaim made without checking the actual real call graph. Rechecked via `mcp__code-review-graph__query_graph_tool` (`callers_of`) and direct source reading:

- `IntelligenceEngine.handleSuggestionTrigger` (the "auto-trigger" path my rapid-fire trace drove via `__e2e__:manual-ask`) has exactly ONE caller in the entire codebase: `ipcHandlers.ts:10883`, which is itself INSIDE the `__e2e__:manual-ask` test-only IPC handler. It is never called from any real, non-E2E production code path. There is no live "auto-trigger" mechanism in the current codebase that calls this method from a real transcript-driven detector — the renderer never invokes it either directly or indirectly.
- The ONLY real production trigger for `runWhatShouldISay` is the renderer's `handleWhatToSay()` (`src/components/NativelyInterface.tsx:3386`), wired to the manual button/hotkey, which calls `generateWhatToSay` → IPC `generate-what-to-say` → `intelligenceManager.runWhatShouldISay(...)` directly — it NEVER goes through `handleSuggestionTrigger`, so the cooldown/speculative-reuse logic inside `handleSuggestionTrigger` that my trace exercised is not even in this real call path.
- Critically, `handleWhatToSay()` starts with `if (!tryBeginOverlayAction('what_to_say')) { ...surface "Still finishing the previous answer" and return... }` (line 3387) — a real, airtight single-in-flight-action guard (`overlayActionInFlightRef`, a `Set` ref scoped to the single overlay window instance) that BLOCKS a second press while a prior one is still streaming. Every call site of `handleWhatToSay` (5 confirmed: lines 4869, 5089, 5234, 6038, plus the hotkey wiring) goes through this same single guarded function. There is no way for two real button presses (or a hotkey + a click) to reach `runWhatShouldISay` concurrently in the shipped UI.
- My `traces/golden-trace-rapidfire-desync.mjs`/`-console.mjs` repro fired 3 CONCURRENT `__e2e__:ask` calls, each of which independently calls `im.reset()` BEFORE building its transcript and triggering (`ipcHandlers.ts:10805`) — so the repro was actually exercising 3 racing `reset()` calls (each bumping `currentGenerationId`) feeding into `handleSuggestionTrigger`, a test-only code path with its own test-only race condition, not a reachable production scenario.

**Revised verdict: H8's underlying code pattern (the emit being "ungated" against `currentGenerationId`) is real and could theoretically bite a FUTURE caller that does allow concurrent triggers, but it is NOT currently reachable through any live UI action given the `tryBeginOverlayAction` guard.** No fix is warranted against a currently-unreachable path — per the resolution rule "touches the least code," and per not inventing work not actually needed. This is logged as a defense-in-depth candidate for Phase 4 hardening (the `emit` truly should carry generation identity for robustness against any FUTURE caller that bypasses the renderer's single-flight guard, e.g. a second window, phone-mirror surface, or a future auto-trigger feature), but is REMOVED from the Phase 1 pinned-fix list. This correction itself is the kind of thing R2 anti-thrash logging is for: I nearly built a whole fix session around a non-reproducible-in-production bug because I tested through a test-only code path without first confirming that path's real-world reachability. Lesson applied going forward: before pinning ANY root cause found via `__e2e__:*` handlers, trace the REAL (non-`__e2e__`) call graph for the exercised method, not just assume the E2E harness mirrors a reachable production flow.

### NEW-1: "in order" collides with the DSA traversal pattern, misrouting résumé questions to `coding_question_answer` (CONFIRMED, root cause pinned, NOT YET FIXED)

### NEW-2: "negotiation"-classified questions get ZERO grounding on the WTA/live-overlay path, even for pure factual JD lookups (CONFIRMED, root cause pinned, fix scope needs a decision)

Case C4-002 ("What company is this role at, and what's the compensation range?") returned a false refusal: "The JD does not specify the company name. Regarding compensation, the document does not state a salary range." Both claims are FALSE — the real fixture JD literally opens with "# Job description — AI Product Engineer @ Helio Labs" and states "Compensation range: 175–200k base + meaningful equity."

Root cause, confirmed via console trace: NO "Grounded what-to-answer in candidate profile" log fired for this turn at all (confirmed absent in the console log window around this turn's timestamp) — `candidateProfile` stayed empty. `electron/llm/transcriptQuestionExtractor.ts`'s `classifyType()` (a THIRD, independent classifier from `AnswerPlanner`'s `answerType` and `KnowledgeOrchestrator`'s own `classifyIntentWithContext`) matches the word "compensation" against its `negotiation` pattern (confirmed via direct regex test) and returns `questionType: 'negotiation'`. `IntelligenceEngine.ts`'s `groundable` gate (~line 1148-1155) only allows `identity, profile_detail, behavioral, jd_alignment, general, follow_up` through to `orchestrator.processQuestion()` — `'negotiation'` is excluded, so grounding never runs, even though `AnswerPlanner`'s OWN classifier correctly identified this as `jd_fact_answer` (required layer `['jd']`, confirmed in `IntelligenceAttribution`).

**FIXED AND VERIFIED (iteration 4, same round) — turned out to be a chain of FOUR stacked classifier disagreements, not just one.** Investigating this one root cause led to discovering three more, each blocking the fix from working until all four were addressed:

1. `IntelligenceEngine.ts`'s `groundable` gate excluded `negotiation`-classified questions entirely. **Fix**: added `isNegotiationButActuallyJdFact` — reuses a new exported helper `isJdFactualLookupNotNegotiationAdvice` (in `AnswerPlanner.ts`, built from the SAME `hasJdReferenceCue`/`NEGOTIATION_ADVICE_CUE_RE` signals `resolveJdSourceType` already relies on) to widen `groundable` only for the JD-fact-lookup shape, leaving genuine negotiation-coaching questions on the existing exclusion.
2. Even with (1)'s gate widened, `KnowledgeOrchestrator.processQuestion`'s own `factualRecall` gate explicitly and by-design NEVER sets `factualRecall: true` for `IntentType.NEGOTIATION` (to keep salary-coaching content out of modes that forbid it) — so the orchestrator call succeeded but its result was still discarded downstream. **Fix**: rather than loosen the general `factualRecall` gate (which risks leaking `salaryContext`/negotiation-script coaching content that's unconditionally appended to `result.contextBlock` earlier in the same function, for ANY `NEGOTIATION`-classified turn), added a narrow, EARLIER carve-out in `KnowledgeOrchestrator.ts`: when `intent === NEGOTIATION` AND `isJdFactualLookupNotNegotiationAdvice(question)`, return `maybeGroundedOnlyResult(question)` directly — a clean, no-retrieval, no-coaching JD-only grounding block that already existed for a different code path (the ambiguous-question inclusion-bias branch) and already sets `factualRecall: true` on its own. This bypasses the coaching-content append entirely for this specific shape.
3. Even with (1) and (2), `KnowledgeOrchestrator`'s intent classifier (`IntentClassifier.ts`) has its OWN, THIRD, independent bug: `IDENTITY_DIRECT_PATTERNS` includes bare `'what company'`/`'which company'` (meant for "what company do YOU work at" — the candidate's own employer) which ALSO matches "what company IS THIS ROLE at" (asking about the target role, not the candidate) — and `IDENTITY_DIRECT_PATTERNS` matches are checked BEFORE the scoring logic and unconditionally return `IntentType.INTRO`, which produces only a ~118-char compact identity block, never reaching NEGOTIATION classification at all. **Fix**: added a `JD_ROLE_FRAME_RE` disqualifier (`/\b(this|the)\s+(role|job|position|posting|listing|opening)\b/i`) so the company-identity patterns only fire when the question is NOT JD-framed.
4. Even with (1), (2), (3) all landed, the answer STILL claimed "the JD does not specify a compensation range" for the company-name half but not the salary half — traced to a FOURTH, distinct gap: `ProfileContextBuilder.ts`'s `buildTargetJobBlock` never rendered `StructuredJD.compensation_hint` into the `<target_job>` block at all, even though the field was correctly extracted and stored. **Fix**: added a `Compensation: ${compensation}` line to the rendered sections.

All four verified together, live, via `test/harness/run-benchmark.mjs` case C4-002: "What company is this role at, and what's the compensation range?" now correctly answers "This role is at Helio Labs. The compensation range is 175,000 to 200,000 base salary plus meaningful equity." — zero fabrication, both facts present, matching the real fixture JD exactly. Regression manifest strengthened to require BOTH facts (`"Helio Labs"` AND `"175"`) plus explicit forbidden-phrase checks (`"does not specify"`, `"does not state"`) so a future partial-regression on either half fails the benchmark, not just a full failure.

Also verified: `npm run typecheck:electron` clean after all 4 changes; full `electron/llm/__tests__` suite still 2483/2543 pass (same pre-existing unrelated failures, confirmed no new regressions from any of the 4 changes).

This chain is a strong illustration of why loop.md's Plateau Rule exists ("if the score improves < 2 points across 3 consecutive iterations... re-do a full golden trace") — a SINGLE symptom (one false-refusal benchmark case) required tracing through FOUR independent, uncoordinated classifiers (`transcriptQuestionExtractor.classifyType`, `KnowledgeOrchestrator`'s `classifyIntentWithContext`/`IntentClassifier.ts`, `AnswerPlanner.planAnswer`, and `ProfileContextBuilder`'s rendering layer) before the fix was complete — fixing any 1, 2, or 3 of the 4 would have looked like meaningful progress in the console logs while the benchmark case kept failing.

## 6e. Golden trace on 2 confirmed thesis-benchmark false refusals — root cause PINNED (iteration 6)

Per iteration 5's NEXT ACTION, did a full golden trace (loop.md §2.1 style: raw retrieval inspection, then full-path trace with temporary tagged logs) on 2 of the 8 confirmed false-refusal cases from the thesis benchmark: THESIS-079 (Logitech C920 camera model) and THESIS-094 (480 total dataset episodes).

**Step 1 — raw retrieval inspection** (`traces/golden-trace-thesis-false-refusal.mjs`, using the existing `__e2e__:inspect-retrieval` handler which calls `buildRetrievedActiveModeContextBlockHybrid` directly): confirmed the answer-bearing text ("Logitech C920", "480") IS present in the raw hybrid-retrieval output for both questions (`allMarkersRetrieved: true`). This initially looked like it REFUTED H1/H6 for these 2 cases — retrieval finds the content.

**Step 2 — full manual-chat path trace** (`traces/golden-trace-thesis-refusal-fullpath.mjs`, driving the exact same `streamGeminiChat` surface the real benchmark uses): found the model's answer STILL refuses, and the `docContextBlock` used for the actual answer generation is dramatically SMALLER (2661 / 6779 chars) than what raw retrieval inspection showed (15936 / 16296 chars) for the same questions. This divergence was the key clue: **`EvidenceResolver` — a governed layer that sits ON TOP of the raw hybrid retriever when `contextOsEvidencePackEnabled` is on — was silently discarding the answer-bearing chunk AFTER a wider raw retrieval had already found it.**

**Step 3 — root cause pinned via a temporary tagged trace** (`[TRACE:evidence-selection]`, added to `electron/intelligence/context-os/evidenceSufficiency.ts`'s `selectSmallestSufficientEvidence`, then reverted after confirming — see below): `EvidenceResolver.finalizePack` calls `selectSmallestSufficientEvidence`, which trims the retrieved chunk set down via a **per-answer-shape cap**: `comparison → 6, list → 5, everything else (including 'numeric') → 3`. Live trace confirmed:
- THESIS-079 (`answerShape: 'list'`, cap 5): `eligibleAfterFilter: 6` — one chunk over the cap gets dropped.
- THESIS-094 (`answerShape: 'numeric'`, cap 3): `eligibleAfterFilter: 6` — HALF the eligible chunks get dropped, and the answer-bearing chunk for a specific numeric fact ("480 episodes") apparently ranks outside the top 3 by the composite `answerRelevanceScore` (distinctive-term coverage + answer-shape density), even though it IS retrievable and IS eligible.

This is a real, confirmed, previously-undocumented root cause: **the cap in `selectSmallestSufficientEvidence` is too aggressive for `'numeric'`-shaped answers on a long (66-page) document with many topically-similar chunks** — the design intent (minimize prompt bloat, avoid noise) is sound, but a fixed cap of 3 for ANY non-list/non-comparison answer shape (which includes `numeric`, `definition`, `methodology`, `follow_up`, and the `general` default) is too small when the retrieval scoring itself has any imprecision in ranking the specific value-bearing chunk near the top.

**This root cause was NOT fixed this iteration** — a cap-tuning change is exactly the kind of "quick patch that might reintroduce a different regression" this campaign's R2 anti-thrash discipline warns against: the cap exists specifically to bound prompt size and reduce noise (per the function's own docstring, "prompt-size safety"), and raising it blindly could reintroduce whatever problem motivated the original 3/5/6 numbers. A responsible fix needs: (a) checking git blame / prior test coverage for why these specific numbers were chosen, (b) either raising the `numeric`/default cap specifically (least invasive) or improving `answerRelevanceScore`'s ranking precision so the right chunk sorts higher even under the existing cap (more invasive, more correct), and (c) re-running the FULL 140-case thesis benchmark (not just these 2 cases) before and after to confirm the fix doesn't regress the 119 currently-passing cases by adding noise. Pinned as the single most concrete, well-evidenced Phase 1 candidate for a focused next iteration.
