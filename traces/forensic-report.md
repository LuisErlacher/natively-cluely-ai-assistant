## 3. Golden Trace — mode+JD/resume grounding (founder symptom #3, H3) — ROOT CAUSE CONFIRMED

### 3a. `_ks_realfixture_verify.mjs` (pre-existing harness, re-run live)

Resume+JD ingested via the dedicated profile pipeline (`__e2e__:ingest-profile-doc`) into a mode switched `general` → `technical-interview` (`sourceAuthority: profile_only`, correctly re-seeded). Question: "Walk me through your most recent project." **Answer was correctly grounded** in the fixture resume's actual "levee" circuit-breaker project (verified: the resume genuinely contains this project verbatim). But `hasTypedEvidencePack: false`, `hasRawCandidateProfile: true` — grounding worked via the LEGACY raw-candidate-profile path, not the new typed `EvidencePack`/`TurnEvidenceCoordinator` machinery (confirming §0's dead-code finding from a different angle: even when grounding works, it isn't going through the new typed infrastructure).

### 3b. NEW trace — JD-only-required question on the SAME resume+JD+technical-interview setup (`traces/golden-trace-jd-fit-legacy.mjs`, `-jd-reqs-repro.mjs`, `-jd-payload-capture.mjs`, `-jd-console-capture.mjs`)

Question: **"What are the key requirements of this role according to the JD?"** — classifies as `answerType: jd_requirements_answer`, whose `AnswerPlanner.ts` definition requires `requiredContextLayers: ['jd']` only (JD-only, resume NOT required — a deliberate design, since this question is about the role, not the candidate).

**Result: the model answered fluently and confidently, but the answer was ENTIRELY FABRICATED — a hallucination, not a refusal.** The real fixture JD (`/tmp/ctxos-fixtures/jd.txt`) is a 3-line JD for "AI Product Engineer @ Helio Labs... LLM prompt engineering, streaming UIs, and Postgres... $175-200k". The model's answer instead invented "distributed systems," "API design," "cloud infrastructure," and "cross-functional teams" — plausible-sounding generic backend-engineer requirements that appear NOWHERE in the actual JD (they're a stylistic echo of the unrelated resume). This is a **direct violation of R5 (zero-hallucination is release-blocking)**, caught on a doc-grounded-adjacent (JD-source) answer type.

**Provider payload capture** (`golden-trace-jd-payload-capture.mjs`, using the new `providerPayloadCapture.ts` infra) proved the literal prompt sent to `gemini-3.1-flash-lite` contained **zero JD evidence text** — the `<transcript>` block held only `[INTERVIEWER]: what are the key requirements of this role according to the jd?`, nothing else. The `answer_contract` header correctly declared `requiredContextLayers: jd`, but no JD content was ever injected to satisfy that requirement. The model, given a shape to fill and no evidence, filled it with invented-but-plausible content.

**Root cause, confirmed via temporary tagged trace logs (`[TRACE:jd-turnsourcedecision]`, `[TRACE:jd-outer-gate]`) added to and rebuilt in `electron/IntelligenceEngine.ts`, then reverted before commit** (see git diff below — these are diagnostic-only, not a fix):

```
[TRACE:jd-turnsourcedecision] {"hasTurnSourceDecision":true,"outcome":"explicit_granted","allowedEvidenceKinds":["profile_jd"],"requiredEvidenceKinds":["profile_jd"],"owner":"profile","wtaDecisionAllowsCandidateProfile":false}
[TRACE:jd-outer-gate] {"hasOrchestrator":true,"isKnowledgeMode":true,"documentGroundedCustomModeActive":false,"wtaDecisionAllowsCandidateProfile":false}
```

The canonical `TurnSourceDecision` engine (the same "Knowledge Source canonical gate" machinery from commits `41edd51`/etc.) correctly resolved this turn: `outcome: 'explicit_granted'`, `owner: 'profile'`, `allowedEvidenceKinds: ['profile_jd']` — it KNOWS the JD is the right and only required source. But `electron/IntelligenceEngine.ts` (the **live WTA/meeting-overlay path**, NOT the manual-chat path) computes `wtaDecisionAllowsCandidateProfile` at line ~1108-1116 as:

```ts
wtaDecisionAllowsCandidateProfile = _wtaTurnSourceDecision.outcome === 'default'
    || _wtaTurnSourceDecision.outcome === 'explicit_granted';
if (_wtaTurnSourceDecision.allowedEvidenceKinds.length > 0) {
    wtaDecisionAllowsCandidateProfile = wtaDecisionAllowsCandidateProfile
        && (_wtaTurnSourceDecision.allowedEvidenceKinds.includes('profile_resume')
            || _wtaTurnSourceDecision.allowedEvidenceKinds.includes('projects'));
        // ^ NEVER checks 'profile_jd' — a JD-only-granted turn always fails this AND
}
```

`allowedEvidenceKinds` is `['profile_jd']` only (no `'profile_resume'`, no `'projects'`) — a JD-only grant, exactly as intended by the canonical decision engine. But the AND-check only recognizes `profile_resume`/`projects`, never `profile_jd`, so `wtaDecisionAllowsCandidateProfile` computes to **`false`** despite the decision engine explicitly granting JD access. This single boolean gates the ENTIRE candidate/JD evidence-fetch block in `IntelligenceEngine.ts` (`if (orchestrator?.isKnowledgeMode?.() && !documentGroundedCustomModeActive && wtaDecisionAllowsCandidateProfile) { ... }`), so `orchestrator.processQuestion()` is never even called, `candidateProfile` stays `''`, and the model gets the answer-shape instructions with zero grounding evidence. `IntelligenceAttribution` confirms: `"structured_jd_used":false,"knowledge_orchestrator_used":false"`.

**This is the exact same bug PATTERN already found and fixed once, on a DIFFERENT code path.** `electron/ipcHandlers.ts` line ~1611-1613 (`_contractAllowsProfile`, the manual-chat equivalent gate) has an explicit code comment dated "Evidence-execution-repair (2026-07-11)" describing this EXACT defect — "this gate previously checked only profile_resume/profile_project, never profile_jd... Confirmed live: 'Does the JD prove that I have Tableau experience?' and 'According to the JD, what are the main responsibilities?' both leaked JD content into a reference_files_only turn through this exact gap" — and fixes it by adding `|| allowsEvidence(turnContract, 'profile_jd')`. That fix landed on the manual-chat path only. **`IntelligenceEngine.ts`'s WTA/live-overlay path — which is what the founder's meeting-overlay actually uses — still has the unfixed, narrower gate.** A SECOND, structurally identical instance of the same unfixed pattern exists at `IntelligenceEngine.ts:2013` (the profile-answer repair gate: `allowsEvidence(wtaTurnContract, 'profile_resume') || allowsEvidence(wtaTurnContract, 'profile_project')` — also missing `profile_jd`).

### Verdict: **H3 CONFIRMED** — pinned root cause, ranked #1 by expected impact.

This is squarely inside R2's "disconnected subsystems" pattern (H9 overlaps): the SAME defect was fixed on one of two parallel implementations of the same authorization gate and not the other. It directly explains:
- Founder symptom #3 (mode+knowledge combination broken) for any JD-only-required question type (`jd_requirements_answer`, `jd_summary_answer`, `jd_fact_answer` per `AnswerPlanner.ts`'s `requiredContextLayers` table) asked through the live overlay/auto-trigger path.
- Founder symptom #2 (nondeterminism/hallucination) — the model doesn't refuse (no visible "I don't have that"), it CONFABULATES a plausible-sounding wrong answer, which is worse than a refusal and is a direct R5 violation.
- This is NOT the "ignoring uploaded sources, refuses citing uploaded-sources-only" framing the founder used verbatim — it's the inverse-but-related failure: the model doesn't refuse, it fabricates. Both symptoms share the same underlying cause class (evidence-injection gate wrongly zeroing out available evidence), just manifesting differently depending on whether the answer-shape instructions leave room for confident invention (JD requirements — yes) vs. explicit refusal templates (identity/behavioral — the FIVE numbered "accuracy_admissions" templates in the system prompt, which is why identity-type questions on this same path DID ground correctly per §3a).

## 4. Anti-thrash note

Per R2: this is technically fixing "the same pattern" that `ipcHandlers.ts` already fixed — but on a DIFFERENT file/code path that was never touched by that fix. This is not re-fixing an already-fixed defect (the manual-chat path's fix is untouched and still correct); it's applying the SAME fix to a sibling code path that shares the bug but was out of scope for the original fix. Per R2 this is legitimate — "you may never fix the same pattern twice" means on the same site, not "never apply the same correct pattern to a second broken site."
