--- Raw model answer ---
Likely Cause:
In a real-time Kafka pipeline, the hardest issues usually come from consumer lag silently growing, partition rebalances during a deploy, or skewed data causing a single task to hot-spot. If the symptom was a backlog with no obvious consumer errors, suspect uneven partition assignment combined with a slow downstream sink, not a bug in the source.

How I Would Investigate:
Start by checking consumer group offsets in Kafka to see which partition is lagging. Pull Flink or consumer task metrics for that specific subtask. Then trace the data path on the slow side: deserialize, transform, sink. Compare throughput across partitions, look at the Watermarks if Flink, and tail logs plus thread dumps on the lagging task. Correlate with any deploy or autoscaling event around the time lag started climbing.

Fix:
Rebalance the workload. Either add more parallel consumers on the hot partition, fix the skewed key with a salting prefix, or batch and tune the sink connection pool. If it was a rebalance storm from a long GC pause, shorten stop-the-world time, give consumers more time between heartbeats, and roll the deploys in smaller batches.

Validation:
Watch the per-partition lag drop back to near zero in the metrics dashboard. Confirm the slowest subtask throughput matches the others within a normal range. Run a load test replaying the same traffic profile to confirm the fix holds before closing the incident.

Prevention:
Add alerting on per-partition lag, not just total lag. Track rebalance counts and 99th percentile processing time per subtask. Keep a runbook for skewed keys and slow sinks, and make any capacity change visible in the dashboard so the next person sees the same picture fast.
